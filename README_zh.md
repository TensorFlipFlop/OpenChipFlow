# OpenChipFlow

> **⚠️ 注意：项目开发中**
> OpenChipFlow 目前正处于开发阶段。受限于目前的 AI Token 额度配给，开发进度相对缓慢，**暂时还不提供给普通用户使用**。我们预计在 **4月中旬** 完成设计收尾，并面向社区用户正式开源使用。

OpenChipFlow 是一款由 AI 驱动的开源芯片设计流程。它旨在通过系统化地调度大型语言模型（LLM），实现从需求规格（Spec）读取、架构规划到 RTL 代码落地、测试平台搭建以及最终闭环验证的全自动化。

## 核心工作模式

OpenChipFlow 目前抽象了三种主要的工作模式，以适应“从0到1的生成”以及“不同模型阶段间的交接工作”：

### 1. Spec Flow (需求驱动模式)
由纯文本的需求规格（Spec）直接驱动的端到端模式。包含以下递进步骤：
- **Precheck（环境检查）**：校验运行环境及依赖项是否就绪。
- **Plan（规划）**：AI 读取并深入分析 Spec，做顶层规划，生成详细的落地设计文档和验证文档。
- **Implement（落地执行）**：根据前阶段打磨好的文档，具体编写 RTL 逻辑代码与 Testbench 验证代码。
- **Verify（验证）**：执行验证代码与仿真环境，并由 AI 自动读取和分析验证结果日志。
- **Regress（回归）**：执行全量的回归测试集，汇总分析结果并确保设计的完备性。

### 2. Handoff Intake (交接准入审查模式)
由外部注入的“交接文档（Handoff）”驱动。Handoff 可以是用户手动生成的产物，也可以是前置 AI（或外部工具）预生成好的代码和更改计划。
- **合规性校验**：系统自动检索和审查交接文件夹内的内容是否合乎规范。
- **反馈闭环**：如果不满足向下游推进的条件，该模式会输出精准的更改建议（即：供 AI 继续修改使用的 prompt）。
- **打磨迭代**：循环往复，不断自动修正，直到该 Handoff 包完全 Pass 审查。

### 3. Verify-Ready Handoff (就绪交接执行模式)
专门对接那些在 `Handoff Intake` 模式下已经审查 Pass 的交接包。
- 以合规的交接文件夹作为直接输入。
- 免去前期的需求分析阶段，直接代入到后续的 **Implement（实施）** 及 **Verify / Regress（验证与回归）** 闭环流程中，加速完成设计的收尾和打磨。
